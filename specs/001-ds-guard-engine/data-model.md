# Data Model: DS-Guard — Design System Enforcement Engine

**Branch**: `001-ds-guard-engine` | **Date**: 2026-06-08

## Entities

### 1. DefinitionDirectory

The authoritative per-system source of truth. Lives in the design system's repository.
Scaffolded by `/dsguard:init`; never generated by the engine itself.

```
DefinitionDirectory
  tokens/
    primitives.json    # DTCG token file — raw values (e.g. colour hex, px sizes)
    semantic.json      # DTCG token file — semantic aliases referencing primitives
  rules.json           # RulesDocument (see below)
  references/          # Optional per-component contracts (markdown)
    <ComponentName>.md
```

**Validation rules**:
- `rules.json` MUST be present and MUST validate against `rules-schema.json`
- At least one token file MUST exist under `tokens/`
- A missing required field in `rules.json` is a hard error (no partial generation)
- Conflicting rules (e.g. `shadows.policy: "none"` with non-empty `shadows.allowed`) are
  a hard error

**Lifecycle**: Created once by maintainer; updated as design evolves; triggers regeneration.

---

### 2. RulesDocument (`rules.json`)

The enforceable policy for one design system. All fields except `name` and `schemaVersion`
are optional — the checker skips a rule class if the corresponding field is absent.

```typescript
RulesDocument {
  name: string             // System identifier, e.g. "acme" — used in generated artifact names
  schemaVersion: string    // Semantic version of the rules schema, e.g. "1.0.0"
  tokenSetComplete?: boolean      // true → undefined-but-plausible values are hard failures (FR-012)

  casing?: "sentence" | "title" | "uppercase" | "none"

  textAlign?: {
    banned?: string[]             // e.g. ["justify"]
    centerOnlyShort?: boolean     // center text only in short (< 3 words) contexts
  }

  type?: {
    scaleOnly?: boolean           // only type-scale tokens permitted; raw px forbidden
    bodyMinPx?: number            // minimum body text size in px
  }

  shadows?: {
    policy: "allowlist" | "none" // "none" = no shadows permitted
    allowed?: string[]           // token names, only relevant when policy = "allowlist"
  }

  radius?: {
    allowed: string[]             // token names of permitted border radii
  }

  fonts?: {
    ui: string                    // font role for UI text
    editorial?: string            // font role for editorial/marketing text
    editorialContexts?: string[]  // prose descriptions of valid editorial contexts
  }

  elevation?: {
    allowed: string[]             // token names of permitted elevation values
  }

  motion?: {
    allowed: string[]             // token names of permitted animation/transition values
  }

  arbitraryAllowlist?: string[]   // values that are explicitly permitted despite not matching
                                  // any token (e.g. "max-w-[72ch]")

  freeformRules?: string          // Markdown guidance conveyed to agent; not checker-enforced
}
```

### 3. DTCGTokenFile

A DTCG-format JSON file. Values are referenced by the generator to populate `CheckerConfig`.

```
DTCGTokenFile
  $schema?: string          // Optional: DTCG JSON schema URI
  [groupName]: {
    [tokenName]: {
      $value: string | number    // The token's value
      $type?: string             // e.g. "color", "dimension", "fontFamily"
      $description?: string
    }
  }
```

**Relationships**: `DefinitionDirectory` contains 1..n `DTCGTokenFile`s.

---

### 4. CheckerConfig (`checker-config.json`)

Generated from the definition. Contains only per-system permitted sets; no engine logic.
Loaded by `checker.mjs` at runtime.

```typescript
CheckerConfig {
  systemName: string             // From rules.json `name`
  schemaVersion: string          // Rules schema version used during generation
  generatedAt: string            // ISO-8601 timestamp
  tokenSetComplete: boolean      // Copied from rules.json; defaults to false

  approvedTokenNames: string[]   // Every defined token's dotted name (e.g. "color.brand.teal");
                                 // used by the checker to detect plausible-but-undefined token
                                 // references in authoritative mode (FR-012)
  approvedColors: string[]       // Resolved from token files (all $type: "color" values)
  approvedTypeScale: string[]    // Resolved from token files (all $type: "dimension" font sizes)
  approvedSpacing: string[]      // Resolved from token files (all $type: "dimension" spacing)
  approvedRadius: string[]       // From rules.json radius.allowed (token names + resolved values)
  approvedShadows: string[]      // From rules.json shadows.allowed (token names + resolved values)
  approvedElevation: string[]    // From rules.json elevation.allowed
  approvedMotion: string[]       // From rules.json motion.allowed
  approvedFonts: {
    ui: string
    editorial?: string
    editorialContexts?: string[]
  }

  bannedTextAlignments: string[] // From rules.json textAlign.banned
  centerOnlyShort: boolean       // From rules.json textAlign.centerOnlyShort
  casing: string                 // From rules.json casing
  typePolicy: {
    scaleOnly: boolean
    bodyMinPx?: number
  }
  arbitraryAllowlist: string[]   // From rules.json arbitraryAllowlist

  freeformRules: string          // Passed through to guidance skill; not used by checker
}
```

**Invariant**: Generated from the same definition always produces byte-identical output
(idempotency). The `generatedAt` field is excluded from idempotency comparison.

### 5. GuidanceSkill (generated markdown file)

A Claude Code skill file. The trigger summary section is always loaded (minimal context cost);
the body is loaded on demand when prompt keywords match.

```
GuidanceSkill (markdown)
  ---
  name: <system>-design-system
  trigger:
    keywords: [list of UI-work keywords derived from rules + token names]
    summary: "<system> design system: [one-line summary of key constraints]"
  ---

  # <System> Design System — Guidance

  ## Approved Vocabulary
  [token names by category: colours, type scale, spacing, shadows, radius, elevation, motion]

  ## Enforceable Rules
  [per rule class: what is permitted, what is banned]

  ## Free-Form Rules
  [freeformRules content verbatim]

  ## Component Contracts
  [links to or content of references/<Component>.md files]
```

**Relationships**: One GuidanceSkill per per-system plugin; generated from CheckerConfig
and definition.

---

### 6. CheckResult

Produced by `checker.mjs` for each finding. Aggregated into a CheckReport.

```typescript
CheckResult {
  file: string           // Relative path to source file
  line: number           // 1-indexed line number
  column?: number        // 1-indexed column (best-effort)
  severity: "ERROR" | "warn"
  ruleClass: string      // e.g. "color", "typeScale", "shadow", "casing", "textAlign"
  message: string        // Human-readable description
  value: string          // The offending value found in source
}
```

---

### 7. CheckReport

The aggregated output of a checker run.

```typescript
CheckReport {
  systemName: string
  checkedAt: string        // ISO-8601 timestamp
  targetPath: string       // Path that was checked
  findings: CheckResult[]
  errorCount: number       // Count of severity = "ERROR"
  warnCount: number        // Count of severity = "warn"
  passed: boolean          // true iff errorCount === 0
}
```

---

### 8. RoutingManifest (`ds-guard.config.json`)

Optional. Present at the consuming project root when multiple design systems are active.

```typescript
RoutingManifest {
  schemaVersion: string
  systems: Array<{
    name: string             // System identifier (matches rules.json `name`)
    pluginDir: string        // Path to the per-system plugin directory
    patterns: string[]       // Path globs; first match in array order wins
  }>
}
```

**Routing rules**:
- Files are matched against `patterns` in system-array order; first matching system wins
- A file matching no pattern is skipped by the checker (no error)
- A file matching multiple patterns in the same system uses the first matching pattern

## Entity Relationships

```
DefinitionDirectory ──contains──► DTCGTokenFile (1..n)
DefinitionDirectory ──contains──► RulesDocument (exactly 1)
DefinitionDirectory ──contains──► ComponentContract (0..n, markdown)

DefinitionDirectory ──generates──► CheckerConfig (1)
DefinitionDirectory ──generates──► GuidanceSkill (1)
DefinitionDirectory ──generates──► PerSystemPlugin (1)

PerSystemPlugin ──contains──► CheckerConfig (1)
PerSystemPlugin ──contains──► GuidanceSkill (1)
PerSystemPlugin ──contains──► checker.mjs (frozen copy)
PerSystemPlugin ──contains──► GateHook (1)

checker.mjs + CheckerConfig ──produces──► CheckReport (per run)
CheckReport ──contains──► CheckResult (0..n)

RoutingManifest ──references──► PerSystemPlugin (1..n)
```
